97 /**
98  * @defgroup Thread 线程管理
99  * @brief 线程管理接口
100  *
101  * RT-Thread操作系统是基于线程调度的多任务系统。
102  * - 调度过程是一种完全抢占式的基于优先级的调度算法。
103  * - 支持8/32/256优先级，其中0表示最高，7/31/255表示最低。最低优先级7/31/255优先级用于空闲线程。
104  * - 支持以相同优先级运行的线程。 共享时间片循环调度用于这种情况。
105  * - 调度器选择下一个已就绪的有最高优先级的线程运行。
106  * - 线程管理有四种状态
107  * - ＃初始状态（还未开始运行的状态）
108  * - ＃运行中/就绪
109  * - ＃挂起
110  * - ＃已结束
111  * - 系统中的线程数不受限制，仅与RAM大小有关。
112  */

129 /**
130  * @defgroup IPC 线程间通讯
131  * @brief 线程间通讯设施
132  *
133  * RT-Thread操作系统支持传统的信号量和互斥量。
134  * - 互斥对象使用继承的优先级来防止优先级反转。
135  * - 信号量释放操作对中断服务程序是安全的。
136  *
137  * 此外，线程获取信号量或互斥锁的阻塞队列可以按优先级或FIFO排序。有两个标志来表示这种机制。
138  * - RT_IPC_FLAG_FIFO
139  * 当资源可用时，最先在此资源上挂起的线程将获得资源。
140  * - RT_IPC_FLAG_PRIO
141  * 当资源可用时，具有最高优先级的线程将获得资源。
142  *
143  * RT-Thread操作系统支持事件/快速事件，邮箱和消息队列。
144  * - 事件机制通过设置一个或多个相应标志位唤醒thead事件发生时的二进制数位。
145  * - 快速事件支持事件线程队列。一旦发生一位事件，可以及时得找到相应阻塞的线程，然后唤醒。
146  * - 在邮箱中，邮件长度固定为4字节，这比消息队列更有效。
147  * - 通信设备的发送动作对于中断服务程序也是安全的。
148  */

161 /**
162  * @defgroup MM 动态内存的管理
163  * @brief 动态内存堆的API接口
164  *
165  * 动态内存管理是一个真实的堆内存管理模块，可以在当前资源满足的情况下，根据用户
166  * 的需求分配任意大小的内存块。而当用户不需要再使用这些内存块时，又可以释放回堆中
167  * 供其他应用分配使用。RT-Thread 系统为了满足不同的需求，提供了两套不同的动态内存
168  * 管理算法，分别是小堆内存管理算法和 SLAB 内存管理算法。
169  *
170  * 小堆内存管理模块主要针对系统资源比较少，一般用于小于2MB内存空间的系统；而SLAB
171  * 内存管理模块则主要是在系统资源比较丰富时，提供了一种近似多内存池管理算法的快速
172  * 算法。两种内存管理模块在系统运行时只能选择其中之一或者完全不使用动态堆内存管理
173  * 器。这两种管理模块提供的API接口完全相同。
174  *
175  * 除上述之外，RT-Thread还有一种针对多内存堆的管理机制，即memheap内存管理。memheap方法
176  * 适用于系统存在多个内存堆的情况，它可以将多个内存“粘贴”在一起，形成一个大的内存堆，
177  * 用户使用起来会感到格外便捷。
178 
179  */

182 /**
183  * @defgroup bsp 中断管理
184  *
185  * @brief 中断管理接口
186  *
187  * RT-Thread的中断管理功能主要是管理中断设备、中断服务例程、中断嵌套、中断栈的维护、
188  * 线程切换时的现场保护和恢复等。
189  *
190  * 当CPU正在处理内部数据时，外界发生了紧急情况，要求CPU暂停当前的工作转去处理这个异步事件。
191  * 处理完毕后，再回到原来被中断的地址，继续原来的工作，这样的过程称为中断。实现这一功能的
192  * 系统称为中断系统，申请CPU中断的请求源称为中断源，当多个中断源同时向CPU请求中断时，就存在
193  * 一个中断优先权的问题。通常根据中断源的优先级别，优先处理最紧急事件的中断请求源，即最先
194  * 响应级别最高的中断请求。
195  *
196  * 中断处理的一般过程如下图所示：
197  * @image html int.png "中断处理过程"
198  *
199  * 当中断产生时，CPU将按如下的顺序执行：
200  * - （1）保存当前处理机状态信息；
201  * - （2）载入异常或中断处理函数到PC寄存器；
202  * - （3）把控制权转交给处理函数并开始执行；
203  * - （4）当处理函数执行完成时，恢复处理器状态信息；
204  * - （5）从异常或中断中返回到前一个程序执行点。
205  *
206  * 中断使得CPU可以在事件发生时才进行处理，而不必让CPU连续不断地查询是否有相应的事件发生。
207  */

264  * @defgroup semaphore 信号量
265  * @brief 信号量
266  *
267  * 信号量是一种轻型的用于解决线程间同步问题的内核对象，线程可以获取或释放它，
268  * 从而达到同步或互斥的目的。信号量就像一把钥匙，把一段临界区给锁住，只允许
269  * 有钥匙的线程进行访问：线程拿到了钥匙，才允许它进入临界区；而离开后把钥匙
270  * 传递给排队在后面的等待线程，让后续线程依次进入临界区。
271  *
272  * 信号量是一种非常灵活的同步方式，可以运用在多种场合中。形成锁，同步，资源计数
273  * 等关系，也能方便的用于线程与线程，中断与线程的同步中。
274  */

276  /**
277  * @defgroup mutex 互斥量
278  * @brief 互斥量
279  *
280  * 互斥量又叫相互排斥的信号量，是一种特殊的二值性信号量。它和信号量不同的是，
281  * 它支持互斥量所有权、递归访问以及防止优先级翻转的特性。
282  *
283  * 互斥量的使用比较单一，因为它是信号量的一种，并且它是以锁的形式存在。
284  * 在初始化的时候，互斥量永远都处于开锁的状态，而被线程持有的时候则立刻转为
285  * 闭锁的状态。互斥量更适合于：
286  *
287  * - 线程多次持有互斥量的情况下。这样可以避免同一线程多次递归持有而造成死锁的问题；
288  * - 可能会由于多线程同步而造成优先级翻转的情况；
289  */

291  /**
292  * @defgroup event 事件
293  * @brief 事件
294  *
295  * 事件主要用于线程间的同步，与信号量不同，它的特点是可以实现一对多，多对多的同步。
296  * 事件另外一个特性是，接收线程可等待多种事件，即多个事件对应一个线程或多个线程。
297  * 同时按照线程等待的参数，可选择是 “逻辑或” 触发还是 “逻辑与” 触发。
298  *
299  * RT-Thread 定义的事件有以下特点：
300  *
301  * - 事件只与线程相关，事件间相互独立：每个线程拥有 32 个事件标志，采用一个 32 bit
302  * 无符号整型数进行记录，每一个 bit 代表一个事件。若干个事件构成一个事件集；
303  * - 事件仅用于同步，不提供数据传输功能；
304  * - 事件无排队性，即多次向线程发送同一事件 (如果线程还未来得及读走)，其效果等同于只
305  * 发送一次。

335  * @defgroup Signal 信号
336  * @brief 信号
337  *
338  * 软中断信号（signal，又简称为信号）用来通知线程发生了异步事件。在软件层次上是对中断机制的一种模拟，
